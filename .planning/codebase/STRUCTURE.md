## Directory Purposes

**`app/(admin)/admin/`:**
- Purpose: Admin-only pages for managing visits, volunteers, mods, and recurring schedules
- Contains: Page shells (server components) + co-located interactive client components (e.g., `visits-admin.tsx`, `volunteers-table.tsx`)
- Key files: `app/(admin)/layout.tsx` (admin shell layout)

**`app/(auth)/`:**
- Purpose: Unauthenticated pages — login form and volunteer registration form
- Contains: `login/page.tsx`, `register/page.tsx`
- Key files: `app/(auth)/layout.tsx` (minimal auth layout)

**`app/(volunteer)/`:**
- Purpose: Authenticated volunteer-facing pages
- Contains: Dashboard, schedule browsing, personal visit history
- Key files: `app/(volunteer)/layout.tsx`, `app/(volunteer)/schedule/schedule-list.tsx`

**`app/api/`:**
- Purpose: All backend logic — REST-style route handlers
- Contains: One `route.ts` per resource/action; no shared router or controller layer
- Pattern: Admin routes under `app/api/admin/`, volunteer routes at top level

**`components/ui/`:**
- Purpose: Low-level UI primitives following shadcn/ui pattern
- Contains: Unstyled-base components wrapped with Tailwind variants
- Generated: Manually added (not auto-generated)

**`lib/`:**
- Purpose: Shared server-side utilities imported by API route handlers
- Contains: Database client singleton, email helpers, class utility

**`prisma/`:**
- Purpose: Database schema definition and seed data
- Key files: `prisma/seed.ts` (run via `prisma db seed`)

**`types/`:**
- Purpose: Shared TypeScript type declarations
- Contains: Custom types not generated by Prisma

## Key File Locations

**Entry Points:**
- `app/layout.tsx`: Root HTML layout, loaded for every page
- `app/page.tsx`: Public landing page at `/`
- `middleware.ts`: Runs before every page render — auth and RBAC gate
- `auth.ts`: NextAuth config — credential validation, session shape

**Configuration:**
- `next.config.ts`: Next.js config
- `tailwind.config.ts`: Tailwind content paths and theme
- `tsconfig.json`: TypeScript config with `@/` path alias pointing to project root
- `docker-compose.yml`: Local PostgreSQL service
- `.env.example`: Documents all required env vars

**Core Logic:**
- `lib/prisma.ts`: Import this for all database access — `import { prisma } from "@/lib/prisma"`
- `lib/email.ts`: Import for sending transactional emails — `import { sendApprovalEmail, ... } from "@/lib/email"`
- `app/api/signups/route.ts`: Signup creation with capacity + duplicate enforcement
- `app/api/admin/volunteers/[id]/route.ts`: Volunteer approval/rejection with county scoping

**Testing:**
- No test files present

## Naming Conventions

**Files:**
- Page entry points: `page.tsx` (Next.js convention)
- API handlers: `route.ts` (Next.js convention)
- Layout wrappers: `layout.tsx` (Next.js convention)
- Co-located client components: kebab-case descriptive name, e.g., `visits-admin.tsx`, `schedule-list.tsx`, `create-visit-form.tsx`
- Shared components: kebab-case, e.g., `nav.tsx`
- UI primitives: kebab-case matching component name, e.g., `button.tsx`, `badge.tsx`

**Directories:**
- Route groups: parenthesized lowercase, e.g., `(admin)`, `(auth)`, `(volunteer)`
- Dynamic segments: bracket notation, e.g., `[id]`, `[...nextauth]`
- Feature directories: kebab-case, e.g., `my-visits`

**TypeScript:**
- Path alias `@/` maps to project root — use for all imports (e.g., `@/lib/prisma`, `@/auth`, `@/components/ui/button`)

## Where to Add New Code

**New Admin Page:**
- Page: `app/(admin)/admin/{feature}/page.tsx`
- Interactive client component: `app/(admin)/admin/{feature}/{feature}-manager.tsx`
- API routes: `app/api/admin/{feature}/route.ts` and `app/api/admin/{feature}/[id]/route.ts`

**New Volunteer Page:**
- Page: `app/(volunteer)/{feature}/page.tsx`
- Interactive client component: `app/(volunteer)/{feature}/{feature}-list.tsx` or similar
- API route: `app/api/{feature}/route.ts`

**New API Endpoint:**
- Create `app/api/{resource}/route.ts` for collection endpoints (GET, POST)
- Create `app/api/{resource}/[id]/route.ts` for item endpoints (GET, PATCH, DELETE)
- Always call `auth()` at the top and check session before any data access
- Define a Zod schema inline for all request bodies

**New Shared UI Component:**
- Primitive: `components/ui/{component-name}.tsx`
- Feature-level shared component: `components/{component-name}.tsx`

**New Library Utility:**
- `lib/{utility-name}.ts`

**Database Changes:**
- Schema: `prisma/schema.prisma`
- Seed additions: `prisma/seed.ts`
